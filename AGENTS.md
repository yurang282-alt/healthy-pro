「以第一性原理！从原始需求和问题本质出发，不从惯例或模板出发。

1. 不要假设我清楚自己想要什么。动机或目标不清晰时，停下来讨论。
2. 目标清晰但路径不是最短的，直接告诉我并建议更好的办法。
3. 遇到问题追根因，不打补丁。每个决策都要能回答“为什么”。
4. 输出说重点，砍掉一切不改变决策的信息。

## 默认执行节奏

1. 小文案、小 UI、局部逻辑调整：先本地修改和本地验证，不急着提交、推送或合入主线。
2. 用户确认效果后，再一次性提交、推送、创建 PR、合入主线和部署。
3. 影响登录、数据、PWA 缓存、构建、部署或线上可用性的改动：即使改动很小，也要单独验证清楚，并建议独立提交，方便回滚和追问题。
4. 任何前端资源或 PWA 相关改动，都要确认构建版本和缓存刷新机制会自动生效，避免用户继续拿到旧页面。
5. 开发过程中优先本地预览和最小必要验证；只有用户需要看线上效果或改动准备发布时，才走 GitHub/Vercel 全流程。
6. 现在产品主入口优先按微信小程序推进，PWA 进入维护模式。小程序发布时，功能、数据、公告和验证优先围绕 `healthy-pro-weapp` 与微信云开发 / CloudBase；PWA 只做必要可用性维护和旧数据对照。
7. 每次发布上线都必须同步整理并发布更新公告：小程序发布优先更新小程序内置公告，并验证「我的 > 更新公告」能看到最新版本；只有当 PWA 仍对外使用、或本次改动影响 PWA 时，才同步更新 PWA 内置公告、Supabase schema seed，并在云端数据库可写时 upsert 到 `public.app_releases`。

## Codex 任务分支与发布

1. 当前 Codex 任务工作区必须始终停留在该任务登记的 `codex/*` 活动分支；不得为了发布在同一工作区切换到 `main`，否则 Codex 会持续提示任务分支与活动分支不一致。
2. 发布到 `main` 前必须先确认 `origin/main` 是当前任务分支的祖先，确保更新可快进且不会覆盖他人提交。
3. 发布 `main` 时优先使用不切换分支的安全 fast-forward 推送；如果必须执行合并操作，使用独立临时 worktree。发布完成后再次确认当前工作区仍在任务登记分支，并将其快进到最新 `origin/main`。
4. 最终交付前必须同时核对：当前任务分支、`origin/main`、工作区清洁状态。不得在当前工作区停留于 `main` 后结束任务。
